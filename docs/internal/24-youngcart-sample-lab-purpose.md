# 24 — 영카트 샘플 Lab: 취지 · 우선순위 · 진행방식

**상태:** 내부 합의 기록 (실행 전 기획 문서)  
**작성:** 2026-07-17  
**관련:** [23-board-community-integration-draft.md](./23-board-community-integration-draft.md) · [영카트5 설치 매뉴얼](https://sir.kr/manuals/yc5/67)  
**대상:** 기획·개발 (관리자 UX 벤치마크 · 커뮤니티 로그인 기초 공유 검토)

> **한 줄:** 영카트/그누보드를 우동공과에 **합치지 않는다.**  
> 로컬에 **분리 샘플**을 띄워 (1) 관리자페이지 UX를 벤치마킹하고, (2) 커뮤니티 로그인 기초정보 공유 가능성만 검토한다.

---

## 1. 왜 하는가

운영자가 이전에 영카트(그누보드 쇼핑몰형) 관리자페이지를 써 본 결과, **UI·디자인·레이아웃·운영 흐름이 매우 편리**했다.

우동공과 관리자페이지를 처음부터 막연히 설계하기보다, **실제 돌아가는 참고 구현체를 옆에 두고** 내부 구조·화면 패턴을 들여다본 뒤, 우리 사이트 관리자에 맞게 재구성하는 것이 목표다.

동시에, 장기적으로는 **그누보드 커뮤니티**와 우동공과가 **로그인·신원(기초 회원 정보)** 을 공유할 수 있는지 학습·검증할 샘플이 필요하다. 이 부분은 [23장 초안](./23-board-community-integration-draft.md)의 「콘텐츠 비공유 · 로그인·신원만 연결」과 같은 방향이다, **당장 DB/인증을 합치지 않는다.**

---

## 2. 목적 우선순위 (잠금)

| 순위 | 목적 | 범위 | 비고 |
|------|------|------|------|
| **1 (주목적)** | **관리자페이지 벤치마킹** | 영카트 `/adm/` UI·레이아웃·메뉴·목록/필터/일괄처리·회원·게시판·상품·주문·환경설정 화면 패턴 | 쇼핑몰 관리 화면도 **관리자 UX 참고용**. 쇼핑몰 기능을 우동공과에 이식하는 것이 아님 |
| **2 (부목적)** | **커뮤니티 로그인 기초정보 공유 검토** | 그누보드 커뮤니티 회원(`g5_member`) ↔ 우동공과 `users` / `user_profiles` 의 **최소 필드·세션·식별** 비교 | 쇼핑몰 회원·주문·결제 데이터는 공유 대상 **아님** |

### 2-1. 주목적에서 볼 것

- 관리자 공통 헤더·사이드바·콘텐츠 영역 레이아웃
- 메뉴 체계와 정보 구조(회원 / 게시판 / 쇼핑몰 / 환경설정)
- 목록·검색·필터·페이지네이션·상태값·일괄 작업 UX
- 상세 편집·권한·설정 화면의 정보 밀도
- 위 패턴을 **우동공과 관리자**에 어떻게 옮길지 (디자인 시스템·home-ui admin과 대조)

### 2-2. 부목적에서 볼 것 (커뮤니티만)

| 우동공과 (정본 후보) | 그누보드 (파생 후보) | 공유 검토 |
|---------------------|---------------------|-----------|
| `users.email` | `mb_email` (+ `mb_id` 정책) | ○ 기초 |
| `users.password_hash` | `mb_password` | △ 해시 상이 → 직접 공유 금지, SSO/재인증 검토 |
| 이름 · 휴대폰 | `mb_name` · `mb_hp` | ○ 기초 |
| 성별 · 생년 · 주소 · 수신동의 | `mb_sex` · `mb_birth` · `mb_addr*` · `mb_mailling`/`mb_sms` | △ 필요 시 |
| 역할(`user_roles`) · 자녀(`students`) | (대응 약함) | ✕ 1차 비공유 |
| 쇼핑몰 주문·장바구니 | `g5_shop_*` | ✕ 벤치마크만, 연동 없음 |

**방향(23장과 동일):** 우동공과 `users`를 정본으로 두고, 그누보드 `g5_member`는 커뮤니티용 파생. 콘텐츠·역할 상세는 공유하지 않는다.

## 3. 하지 않는 것 (경계)

| 금지 | 이유 |
|------|------|
| 영카트를 우동공과 **실서비스에 합치기** | 샘플·학습용. 배포·인증 오염 방지 |
| 기존 `users` / 세션 / OAuth를 영카트 인증으로 **덮어쓰기** | 실서비스 인증 보호 |
| 같은 DB에 `g5_*`와 `users`를 **섞어 설치** | 실수·백업·권한 혼선 |
| 쇼핑몰 결제·주문 데이터를 우동공과와 **연동** | 부목적 범위 밖 |
| 닷홈/카페24/Actions에 샘플 **배포** | 로컬 Lab 전용 |

## 4. 설치·실행 원칙 (진행방식)

상세 설치 절차는 실행 단계에서 체크리스트로 두고, **원칙만** 여기에 고정한다.

| 원칙 | 내용 | 왜 |
|------|------|-----|
| **폴더 분리** | 권장: `_sample/youngcart/` (루트 하위, `public/`·`src/` 밖) | FTP(`public/`,`src/`)·실서비스 docroot와 자연 분리 |
| **런타임 분리** | 기존 API `8080`에 path 끼워 넣지 않음. **별도 포트(예: 8090)** + Apache/PHP 컨테이너 | 현재 API는 `php -S` + `pdo_mysql`만 — 영카트(mysqli·GD·`.htaccess`)와 비호환 |
| **DB 분리** | `youngcart_dev` (기존 `study114_dev`와 별도 DB). prefix `g5_` 유지 | 테이블·백업·오염 방지 |
| **배포 제외** | `.gitignore`에 `/_sample/` · Actions/FTP 경로에 없음 | 운영 사이트에 샘플 유출 방지 |
| **매뉴얼 준수** | [영카트5 설치하기](https://sir.kr/manuals/yc5/67) — `data/` 권한, 설치 마법사, 최고관리자 | 공식 설치 경로로 재현성 확보 |

접속 예 (로컬 Lab):

- 커뮤니티: `http://127.0.0.1:8090/`
- 쇼핑몰(참고): `http://127.0.0.1:8090/shop/` 또는 상단 메뉴
- **관리자(주목적):** `http://127.0.0.1:8090/adm/`

## 5. 설치 이후 작업 순서

```
[Phase A] 샘플 기동
  → Docker + youngcart_dev + 매뉴얼 설치
  → 커뮤니티·쇼핑몰·관리자 화면이 뜨는지 스모크

[Phase B] 주목적 — 관리자 벤치마크 (우선)
  → /adm/ 메뉴 전체 인벤토리
  → 레이아웃·목록·필터·일괄·상세 패턴 분류
  → 우동공과 home-ui admin / 설계 초안과 대조표 작성
  → “가져올 패턴 / 버릴 패턴 / 우리식 재해석” 정리

[Phase C] 부목적 — 커뮤니티 로그인 기초 (후속)
  → g5_member ↔ users/user_profiles 필드 매핑표 확정
  → 세션·쿠키·로그인 진입점 비교 (코드 읽기)
  → SSO/식별자 매핑 후보만 문서화 (구현은 별도 합의 후)
  → 23장 초안과 정합 여부 점검
```

Phase B가 끝나기 전에 Phase C 구현(코드 연동)으로 넘어가지 않는다.
