## 3. 가장 큰 마찰: 로그인 ID

| | 우동공과 | 그누보드 |
|--|----------|----------|
| 사용자가 치는 ID | **이메일 전체** | 짧은 **`mb_id` (최대 20자)** |
| 이메일은 | ID 자체 | `mb_email`에 **별도** 저장 |

이메일은 대개 20자를 넘는다 (`parent@example.com` = 18, 실사용은 더 김).  
→ **`mb_id = email`로 맞추는 방식은 불가에 가깝다.**

### 권장 ID 전략

| 역할 | 값 | 설명 |
|------|-----|------|
| 사용자가 아는 로그인 | **이메일 + 비번** (우동공과와 동일) | UX 단일화 |
| GNU 내부 `mb_id` | `u{users.id}` 또는 `s114_{id}` 등 **짧은 파생키** | 20자 제한·고유성 |
| GNU `mb_email` | `users.email` | 표시·메일·중복검사용 |
| 연결 | `member_external_links` (user_id ↔ mb_id) | 23장 초안과 동일 |

커뮤니티 화면에서 GNU 기본 “아이디로 로그인” 폼을 그대로 쓰지 말고, **우동공과 SSO로 들어온 세션**을 쓰는 편이 맞다.  
(커뮤니티 단독 가입은 막거나, “본서비스에서 가입”으로 유도.)

## 4. 비밀번호 — 어디를 맞출까?

| 방식 | 편의 | 보안·운영 | 판정 |
|------|------|-----------|------|
| 해시 미러링 (한쪽 비번을 다른쪽에 복사) | 구현은 쉬워 보임 | 해시 알고리즘 상이 · 탈퇴/변경 동기화 사고 | **비권장** |
| **우동공과만 비번 보관 + SSO** | 사용자는 비번 하나 | GNU는 더미/랜덤 해시 + 로그인 우회 | **권장** |
| 그누보드를 정본으로 | GNU UX에 익숙 | 본서비스·OAuth·역할 모델을 깨뜨림 | **비권장** |

비밀번호 변경·찾기·소셜 로그인은 **전부 우동공과 auth**에서만.

## 5. “어느 쪽에 맞추는 게 더 편한가?”

### 추천: **그누보드를 우동공과에 맞춘다** (S114 = Master)

| 이유 | 설명 |
|------|------|
| 이미 잠금 | 2·4장 가입·DB가 본서비스 정본 |
| 사용자 여정 | 가입·유료·쪽지·매칭이 전부 우동공과 |
| ID 모델 | 이메일 로그인이 현대 UX·OAuth와 맞음 |
| 역할 | 학부모/공부방/과외는 GNU `mb_level`로 표현 불가 → 상세 비공유가 맞음 |
| 운영 | 차단·탈퇴를 본서비스에서 한 번 하면 커뮤니티만 파생 반영 |
| 샘플 위치 | `_sample/yc5.6.30`은 Lab — 실서비스 스키마를 샘플에 종속시키면 안 됨 |

### 그누보드에 우동공과를 맞추면 생기는 비용

- 로그인 ID를 20자 `mb_id` 규칙으로 바꾸거나 이중 ID UX
- bcrypt/`get_encrypt_string` 이중 관리
- 닉네임 필수·포인트·회원등급 UI를 본가입에 끌려옴
- SSOT 재잠금 비용이 큼

### 그누보드에만 맞춰도 되는 것 (로컬 예외)

| 항목 | 이유 |
|------|------|
| `mb_nick` | 커뮤니티 표시명 — 본서비스에 없어도 됨 |
| `mb_level` 2단 | 마케팅용 단순 등급 (23장) |
| `mb_point` / 스크랩 / 서명 | 커뮤니티 전용 |
| 게시판·댓글 프로필 | 콘텐츠 비공유 원칙 |

## 6. 동기화 방향 (실무 규칙)

```
[가입·수정·탈퇴] 우동공과 users / user_profiles
        │
        ▼  (SSO 또는 webhook/배치)
[파생] g5_member 기초 컬럼만 upsert
        · mb_id = 파생키
        · mb_email, mb_name, mb_hp, mb_sex, mb_birth, 주소·수신 (선택)
        · mb_password = 사용 불가한 랜덤 해시 (직접 로그인 차단)
        · mb_nick = 최초 자동생성 후 GNU에서만 변경 허용(정책 선택)
```

| 이벤트 | 동작 |
|--------|------|
| S114 가입 | GNU 행 생성(또는 첫 커뮤니티 진입 시 lazy create) |
| S114 프로필 수정 | 기초 필드만 GNU 갱신 |
| S114 차단/탈퇴 | `mb_intercept_date` / `mb_leave_date` 반영 |
| GNU에서 이름·전화 수정 | **1차는 금지** (관리자·유저 모두) — 정본 오염 방지 |
| GNU에서 닉만 수정 | 허용 가능 |
