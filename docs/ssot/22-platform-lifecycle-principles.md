## 0. 핵심 한 줄

우동공과의 등록·공개·접촉은 **운영자 심사·심의·승인·반려 없이**, 당사자가 정보를 입력·확인·공개하고, 상대가 보고 판단하며, **접촉은 당사자 간 합의**로 진행한다.

---

## 1. 하지 않는 것 (플랫폼)

| ✕ 금지 | 이유 |
|--------|------|
| 등록 서류·프로필 **심사 후 공개 허가** | 플랫폼이 심사자가 되면 분란·기준 논쟁 |
| 운영자 **승인 / 반려 / 보완 요구** | 「왜 안 되나요?」 민원·공정성 이슈 |
| `pending` = **검토·심사 대기** (공부방·과외 `profile_status`) | §3 |
| 플랫폼이 **상담·매칭을 승인** | 당사자 판단 영역 |

### 1-1. 금지 UI·문구 (등록·공개 lifecycle)

검토중 · 심사 대기 · 반려 · 보완 요청(운영자 투기) · 승인 대기

**권장:** 저장중 · 공개중 · 숨김 · **공개 준비 미완료** · 공개 전 확인 필요 · 상담 중지 · 정원 마감

---

## 2. 대신 하는 것

| ○ 방식 | 설명 |
|--------|------|
| **필수 입력 체크리스트** | 시스템 자동 판정 — 미충족 시 공개 버튼 비활성 |
| **공개 준비 미완료** | DB 상태 ✕ · **UI 계산값** (필수 항목 부족) |
| **자기확인 (confirm)** | 공개 전 「학부모/상대에게 이렇게 보입니다」— **심사 ✕** |
| **당사자 직접 전환** | 공개 · 숨김 · 삭제 · (공급) 상담상태 — **본인 선택** |
| **당사자 합의 접촉** | 쪽지·블라인드 접촉 (16장) |
| **사후 안전장치** | 신고 · 차단 · 안전 가이드 (17장) — **사전 심사 ✕** |

## 3. `profile_status.pending` (공부방·과외) — deprecated

5장·8장 DDL에 `profile_status` enum **`pending`이 존재할 수 있다.**  
**1차 공개 lifecycle에서는 사용하지 않는다** (enum 당장 삭제 ✕ · 마이그레이션 영향 최소화).

| 항목 | 정책 |
|------|------|
| DDL | `pending` **유지** (deprecated) |
| 의미 | **운영자 검수·승인·반려로 사용 금지** |
| 20·21장 UI | **표시·전환·탭에 사용하지 않음** |
| 공개 준비 미완료 | **`pending`이 아님** → 체크리스트 계산 (§2) |
| 1차 lifecycle 축 | `draft` → `published` ↔ `hidden` + `deleted_at` |

**SSOT 주석:** [5장 §4-2](05-study-room-db.md#4-2-profile_status--pending-deprecated) · [8장 §4-1](08-tutor-registration-db.md#4-1-profile_status--pending-deprecated)

> **학생(19장)** 은 `students.exposure_status` (`draft`/`published`/`hidden`/`deleted`) — `profile_status.pending`과 **별개**.

## 5. 본 원칙과 다른 축 (혼동 방지)

| 축 | 관계 |
|----|------|
| **회원 계정** `users.status` 의 `pending` | 계정 활성화 등 — **§3과 별개** |
| **증빙·서류 제출** (`*_verification_documents`) | 당사자 **제출·표시** — **운영자 심사·반려 ✕** · P15-10 = 제출자료 상태만 ([15장 §2-1](15-mypage-structure.md#2-1-p15-10-제출자료-상태) · [21장](21-tutor-registration-management.md)) |

| **신고·제재** | **사후** — §2 |
| **시스템 안내** (15장 §1-1) | 상품 만료 · 공개 준비 미완료 · 제출자료 상태 등 — **운영자 반려 알림 ✕** |

## 7. 용어 정리 (1차)

등록·공개 lifecycle·마이페이지 안내 문구에서 **아래 표현을 쓰지 않는다.** 충돌 시 **22장 우선**.

| ✕ 피할 표현 | ○ 대체 표현 | 비고 |
|-------------|-------------|------|
| 검증 (운영자) | 제출자료 · 제출자료 상태 | 플랫폼 보증 느낌 제거 |
| 인증 (서류 심사) | 본인 확인용 자료 | 실제 인증 API 연동 시만 「확인」 |
| 심사 · 검수 | 공개 전 확인 · 자동 체크 | 체크리스트 계산 |
| 반려 | 공개 준비 미완료 | 운영자 판단 ✕ |
| 승인 (운영자) | 공개 · 공개함 | 당사자 직접 전환 |
| 통과 | 공개 가능 | 체크리스트 충족 |
| 운영 알림 | **시스템 안내** | 15장 §1-1 |
| 검증 반려 | *(삭제)* | 해당 알림 유형 **없음** |

### 7-1. 시스템 안내 — 허용 유형 (1차 방향)

| 안내 유형 | 가능 | 이유 |
|-----------|:----:|------|
| 상품 만료 예정·완료 | ○ | 시스템 자동 |
| 공개 준비 미완료 | ○ | 체크리스트 계산값 |
| 필수항목 누락 | ○ | 시스템 자동 |
| 상담상태 변경 (예: 정원 마감) | ○ | 사용자가 선택한 상태 반영 |
| 제출자료 미제출·상태 요약 | ○ | 제출 여부 표시 |
| 검증 반려 | ✕ | 운영자 판단·반려 느낌 |

### 7-2. §7 적용 범위 · 예외 (혼동 방지)

§7 금지·대체 표는 **등록·공개 lifecycle · 마이페이지 · 사용자-facing 안내**에 적용한다. 아래는 **별개** — §7과 충돌하지 않음.

| 맥락 | 표현 | 비고 |
|------|------|------|
| 폼 **입력 유효성** | 필수값 · 형식 검사 | 14·19·20·21장 · UI 프리뷰 — **운영자 심사 ✕** |
| **회원 인증** (가입) | 이메일·휴대폰 인증 | 2·14장 · auth-ui — **서류 심사 ✕** |
| **사후 조치** | 신고 처리 · 관리자 숨김 | 5장 후기 · 17장 — **사전 lifecycle 심사 ✕** |
| **Prime/Pick 배정** | 운영 슬롯 배정 | 18·11장 — **프로필 공개 심사 ✕** |
| DOC-CHECKLIST | 「SSOT 대조」 | 개발 문서 용어 · 사용자 UI **아님** |
| DB 테이블명 | `*_verification_documents` | 레거시 명칭 · 의미 = **제출자료 저장** (§5) |
