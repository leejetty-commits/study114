## 부록 A. 로그인 화면 구조 (화면 묘사)

### A-1. 한 줄 컨셉

로그인화면은 화려한 랜딩이 아니라, **10초 안에 진입이 끝나는 인증 카드**이면서도, 로그인 후 들어갈 **동네 탐색 홈의 실루엣**이 은근히 느껴지는 화면이다.

### A-2. 레이아웃

```
[상단 미니 바]  지역선택 | 이용안내 | 고객센터

[메인 스테이지]
  흐린 배경: 공부방(지도형) + 과외(조건형) 홈 실루엣
  [중앙 로그인 카드]  로고 · 로그인 · 이메일/비번 · 소셜(네이버→카카오→구글)
  [카드 아래]  목적 칩 3개 · 짧은 안내 · 둘러보기
```

### A-3. 원칙

| 항목 | 잠금 |
|------|------|
| 역할 분기 | 로그인 화면에서 **강하게 요구하지 않음** — 공통 진입 카드 |
| 배경 | 홍보 사진 ✕ · **흐린 홈 실루엣** ○ (클릭 불가 · blur·저대비) |
| 소셜 | 보조가 아니라 **동등한 진입 경로** (한국 3축 순서) |
| 유료/권한 | 전면 설명 ✕ |
| 스크롤 | 긴 랜딩 ✕ · **한 화면 안에서 입력·진입** |

### A-4. 목적 칩 (카드 아래)

- 우리동네 공부방 찾기
- 과외쌤 비교하기
- 쪽지로 이어서 상담하기

### A-5. 문구 톤

| 구분 | 예시 |
|------|------|
| 제목 | 로그인 |
| 보조문구 | 우리 동네 공부방·과외쌤 찾기와 쪽지 연결을 이어서 이용하세요. |
| 회원가입 유도 | 처음이신가요? 회원가입 · 학부모·공부방·과외쌤 **공통 계정** |

## 부록 B. 인증정책 기본 원칙 (표준성·예측 가능성)

차별화보다 **예측 가능성**을 우선한다. 사용자가 새로 학습해야 하는 독특한 규칙보다, 국내 주요 서비스에서 기대하는 **표준 패턴**을 따른다.

| 항목 | 1차 잠금 |
|------|----------|
| 기본 로그인 | **이메일 + 비밀번호** |
| 소셜 | 보조가 아닌 **동등 진입** · 한국 **네이버 → 카카오 → 구글** |
| 계정 구조 | 역할별 별도 로그인 ✕ · **공통 계정 + 역할 확장** ○ |
| 이메일 | 로그인 ID · UNIQUE · trim + **소문자 정규화** |
| 동일 이메일 | 소셜·이메일 **연결 우선** · 자동 병합(다른 이메일) ✕ |
| 로그인 실패 | **이메일 또는 비밀번호를 확인해 주세요** (존재 여부 노출 ✕) |

## 부록 C. Cursor handoff — 1차 인증정책 구현 정리 (§16)

화면 창의성보다 **표준적이고 익숙하지만 단단한 동작 규칙**을 먼저 맞춘다.

### C-1. 회원가입 기본 원칙 (§16-1)

- 1차 회원가입은 **이메일 + 비밀번호 기반 공통 계정 생성**.
- 회원가입 시 **SMS 패스코드/휴대폰 OTP 사용하지 않음**.
- 비밀번호 설정 후 별도 인증번호 단계 없이 **가입 완료 → 자동 로그인**.

```
회원 구분 선택
→ 공통 가입 폼 입력
→ 이메일/비밀번호 검증
→ 계정 생성
→ 자동 로그인
→ 가입 완료
→ 역할별 기본등록 또는 홈 진입
```

### C-2. 이메일 확인 정책 (§16-2)

- 1차에서 **회원가입 직후 전면 이메일 인증 강제하지 않음** (계정 생성·자동 로그인 허용).
- 아래 행동은 **이메일 인증 완료 후에만** 허용:
  - 공개 등록/공개 전환
  - 쪽지 발송 또는 공식 접촉 시작
  - 유료 결제/유료 권한 활성화
  - 계정 복구 정보 변경
- 정책 핵심: **가입 전면 인증**이 아니라 **위험 행동 전 인증**.
- 미인증 시 해당 시점에 **이메일 인증 유도 모달/안내**.

### C-3. 패스코드/OTP 정책 (§16-3)

- **회원가입용 패스코드 없음** · 휴대폰 문자 OTP 1차 제외 (비용·마찰·MVP).
- 인증이 필요할 때는 **이메일 링크 인증** 우선.
- 「비밀번호 설정 후 문자 인증번호」구조는 1차 기본안 **채택하지 않음**.

### C-4. 비밀번호 정책 (§16-4)

#### 기본 규약

| 규칙 | 값 |
|------|-----|
| 길이 | **8자 이상 14자 이하** |
| 조합 | **영문 + 숫자 + 특수문자** 필수 |
| 공백 | 허용하지 않음 |
| 확인 | 비밀번호 확인값과 **완전 일치** |

#### 금칙/차단

- 이메일 아이디(local-part) 그대로 포함 ✕
- 이름 그대로 포함 ✕
- 휴대폰 뒤 4자리 그대로 포함 ✕
- 널리 알려진 취약 패턴 차단 (예: `12345678`, `password1!`, `qwer1234!` 등)
- 동일 문자·숫자 과도 반복 ✕
- 키보드 순서·단순 증가 수열 등 쉬운 패턴 ✕

#### 구현 원칙

- **클라이언트 안내 + 서버 검증** 모두 필수 (프론트만 검사 ✕).
- 입력 필드 아래 규칙 짧게 명시:
  - 8~14자 · 영문, 숫자, 특수문자 포함 · 쉬운 비밀번호 사용 불가

### C-5. 비밀번호 찾기/재설정 (§16-5)

```
비밀번호 찾기 페이지
→ 가입 이메일 입력
→ 재설정 메일 발송
→ 메일 내 일회성 링크 (30분)
→ 새 비밀번호 입력
→ 재설정 완료 → 다시 로그인
```

| 원칙 | 내용 |
|------|------|
| 링크 | **일회성** · 사용 후 무효 · 만료 **30분** 권장 |
| 재설정 완료 | 이전 재설정 토큰 **전부 무효화** |
| 금지 | 임시비밀번호 발급 ✕ · 더미 인증번호(`123456`) ✕ · 외부 증명 없이 재설정 ✕ |

### C-6. 로그인 실패 문구 (§16-6)

- 계정 존재 여부 **직접 노출하지 않음**.
- 기본: **이메일 또는 비밀번호를 확인해 주세요**.

### C-7. 계정 연결/중복 (§16-7)

- 동일 이메일 → 이메일 가입·소셜 **같은 공통 계정으로 연결**.
- 서로 다른 이메일 → **자동 병합하지 않음**.
- 학부모/공부방/과외쌤 → 별도 로그인 계정 ✕ · **공통 계정 아래 역할 확장** ○.

### C-8. 미인증 계정 제한 (§16-8)

| 허용 | 제한 |
|------|------|
| 로그인 · 기본 탐색 | 공개 전환 · 쪽지 발송 · 유료 결제 · 계정 복구 정보 변경 |

→ **읽기/탐색 가능, 공개/접촉/결제는 제한**.

### C-9. Cursor 구현 체크리스트 (§16-9)

1. 회원가입 직후 **자동 로그인**
2. 가입 단계 **SMS OTP 없음**
3. 이메일 인증 = **행동 전 인증**
4. 비밀번호 **8~14자 + 영문/숫자/특수 혼합**
5. 기본 취약 패턴 차단
6. **서버/클라이언트 동일 검증**
7. 비밀번호 찾기 = **이메일 링크 재설정**
8. 로그인 실패 문구 일반형
9. 동일 이메일 소셜/이메일 계정 연결
10. 역할 = 새 계정 ✕ · **공통 계정 확장** ○

### C-10. 한 줄 최종 원칙 (§16-10)

**가입은 짧고 빠르게 통과시키되, 공개·접촉·결제 같은 위험 행동 직전에만 인증 강도를 올린다.**

## 부록 E. 비밀번호 찾기/재설정 화면 (§17)

이 절은 Cursor가 비밀번호 찾기 화면을 임의 해석하지 않도록, **페이지 구성부터 상태 흐름, 문구 톤, 예외 처리까지** 화면 언어로 풀어 쓴 구현 가이드다.

### E-1. 전체 원칙

- 비밀번호 찾기는 **계정 복구의 시작점**이지, 새로운 가입 유도나 서비스 소개 화면이 아니다.
- 화면은 로그인화면과 같은 톤을 유지하되, **더 조용하고 단순**해야 한다.
- 한 화면에는 **한 가지 목적만** 둔다.
- 단계는 아래 4단계로 고정한다.

```
비밀번호 찾기
→ 재설정 메일 발송 완료
→ 메일 링크 클릭 후 새 비밀번호 설정
→ 재설정 완료
```

- 회원가입 때처럼 휴대폰 문자 OTP, 별도 패스코드 입력, 추가 질문 단계는 넣지 않는다.
- 복구 과정에서는 제품 개성보다 **이해 속도, 안정감, 예측 가능성**을 우선한다.

### E-2. 비밀번호 찾기 시작 화면 (`#/find-password`)

| 요소 | 잠금 |
|------|------|
| 상단 | 로고(로그인으로) — 실루엣 **없음** |
| 제목 | 비밀번호 찾기 |
| 보조문구 | 가입한 이메일을 입력하면 비밀번호를 다시 설정할 수 있는 링크를 보내드립니다. |
| 필드 | 라벨 `이메일` · placeholder `가입한 이메일을 입력하세요` |
| CTA | 재설정 메일 보내기 |
| 하단 링크 | 로그인으로 돌아가기 · 회원가입 |

**입력 규칙:** 앞뒤 공백 제거 · 소문자 정규화 · 형식 오류 시 `올바른 이메일 형식이 아닙니다.`

### E-3. 동작 원칙 (계정 존재 비노출)

- 제출 후 **가입 여부와 관계없이 동일한 성공형 화면**으로 전환한다.
- API 응답 문구: `입력한 이메일로 비밀번호 재설정 안내를 보냈습니다. 잠시 후 메일함을 확인해 주세요.`
- 로그인 실패 문구 일반형 원칙(부록 C-6)과 일관된다.

### E-4. 재설정 메일 발송 완료 화면

| 요소 | 잠금 |
|------|------|
| 아이콘 | 체크(조용한 원형) |
| 제목 | 메일을 확인해 주세요 |
| 설명 | 재설정 링크 발송 · 스팸함 확인 · 재시도 안내 (2~3줄) |
| 버튼 | 메일 다시 보내기 · 이메일 다시 입력 |
| 하단 | 로그인으로 돌아가기 |
| 재전송 | **서버·클라이언트 동일 3분(180초)** · 버튼 비활성 + `메일 다시 보내기 (M:SS)` |
| API | `POST /api/auth/password/forgot.php` → `resend_available_in`(초) — 쿨다운 중에도 `ok:true` · 계정 존재 비노출 |

「가입되지 않은 이메일입니다」류 문구 **금지**.

### E-5. 재설정 메일 템플릿

| 항목 | 내용 |
|------|------|
| 제목 | `[우동공과] 비밀번호 재설정 안내` |
| 본문 | 요청 접수 → **HTML 버튼** `비밀번호 다시 설정하기` → **30분 유효** → 무시 안내 |
| 형식 | `multipart/alternative` — plain(링크 텍스트) + HTML(단일 CTA 버튼) |
| 구현 | `src/Auth/PasswordResetMailTemplate.php` |
| 금지 | 마케팅 문구 · 장황한 안내 · 배너 · 버튼 2개 이상 |

로컬 개발: `storage/logs/mail.log` — `--- plain ---` / `--- html ---` 구분 기록 (SMTP 대체). **Dev/QA만** plain 링크 복사로 검증.

### E-5a. validate-token API 해석 (고정)

| 필드 | 의미 |
|------|------|
| `ok: true` | API 호출·처리 자체는 정상 |
| `ok: false` | HTTP/핸들러 실패 (토큰 상태와 무관) |
| `status` | 비즈니스 토큰 상태: `valid` · `invalid` · `expired` · `used` |

⚠️ `ok: true, status: "invalid"` = “요청은 성공, 토큰은 무효”. 프론트는 `preview/auth-ui/src/password-reset-api.js` 계약을 따른다.

### E-6. 새 비밀번호 설정 (`#/reset-password?token=`)

| 요소 | 잠금 |
|------|------|
| 제목 | 새 비밀번호 설정 |
| 보조문구 | 새로 사용할 비밀번호를 입력해 주세요. |
| 필드 | 새 비밀번호 · 새 비밀번호 확인 |
| 규칙 안내 | 3줄 **기본 노출** (접기 ✕) |
| CTA | 비밀번호 변경하기 |

규칙 3줄: `8~14자` · `영문, 숫자, 특수문자 포함` · `쉬운 비밀번호 사용 불가`

진입 시 `GET /api/auth/password/validate-token.php`로 토큰 상태 선검증.

### E-7. 검증·오류 문구

| 상황 | 문구 |
|------|------|
| 길이 | 8~14자 이내로 입력해 주세요 |
| 조합 | 영문, 숫자, 특수문자를 모두 포함해 주세요 |
| 불일치 | 비밀번호가 서로 일치하지 않습니다 |
| 금칙/취약 | 사용할 수 없는 비밀번호입니다. 더 안전한 비밀번호로 다시 입력해 주세요 |

금칙어 상세(이메일·이름·휴대폰 등)는 **사용자에게 노출하지 않음**.

### E-8. 예외 상태 화면

| 상태 | 제목 | CTA |
|------|------|-----|

| 무효 | 유효하지 않은 요청입니다 | 비밀번호 찾기로 돌아가기 |

API `error`: `expired` · `used` · `invalid` — 기술 코드·스택 **노출 금지**.

### E-9. 재설정 완료

- 제목: `비밀번호가 변경되었습니다`

- CTA: `로그인하기`
- **자동 로그인 금지** — 사용자가 새 비밀번호로 직접 로그인.
