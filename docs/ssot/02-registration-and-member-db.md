## 2. 1차 가입방식 잠금

### 2-1. 기본 원칙

```
회원 구분 선택 → 공통 기본정보 가입 → 가입 완료 → (역할별) 상세등록
```

| 원칙 | 내용 |
|------|------|
| 공통 계정 | 역할별 **별도 계정 체계 없음** — 공통 `users` 1건 생성 |
| 짧은 가입 | 처음부터 역할별 긴 폼으로 **모든 정보를 받지 않음** |
| 서비스 참여 | **상세등록 완료 후** 본격 노출·운영 (가입 완료 ≠ 상세등록 완료) |

**화면 순서 (잠금):** 약관동의 → 회원 구분 선택 → 공통 가입 폼 → 가입 완료  
→ 상세: [§3 인증·가입 화면](#3-인증가입-화면-잠금)

### 2-2. 회원 구분 선택

| UI 라벨 | 내부값 | 비고 |
|---------|--------|------|
| **학생(학부모)** | `student` | 소비자 축 |
| **공부방** | `study_room` | 운영자 의향 |
| **과외쌤** | `tutor` | 운영자 의향 |

- 3개 역할 **카드 구조** 유지.
- 긴 역할 설명보다 **가입 주체 빠른 구분**에 집중.
- 선택값은 **주 사용 유형** 표시용 — 실제 처리는 공통 계정.

### 2-3. 학생 축 잠금

| 구분 | 잠금 |
|------|------|
| 화면 표기 | **학생(학부모)** (또는 학생/학부모) |
| 운영 기준 | **학부모(보호자)** |
| 로그인 | 학생 **별도 계정 없음** — `students`는 학부모 소속 |
| UI | 필요 시 「학부모입니다」 체크/표시 **허용** |

### 2-4. 공통 기본정보 가입 폼

역할별 별도 가입 폼 **금지**. 아래 **공통 폼**만 통과.

| UI 라벨 | `name` | 필수 |
|---------|--------|------|
| 이메일(ID) | `email` | Y |
| 비밀번호 | `password` | Y |
| 비밀번호 확인 | `password_confirm` | Y |
| 이름 | `name` | Y |
| 성별 | `gender` | Y |
| 생년월일 | `birth_date` | Y |
| 휴대폰 | `phone` | Y |
| 주소 | `address` | Y |
| 문자 수신 동의 | `sms_consent` | N |
| 이메일 수신 동의 | `email_consent` | N |
| 안전번호 사용 여부 | `safe_number_use` | N |

> Notion 원문 「이메일」 중복 항목은 **이메일(ID) 1개**로 통합 잠금.

### 2-5. 가입 완료 후 상세등록 유도

| CTA | 설명 |
|-----|------|
| `{회원구분} 상세등록` | 역할별 상세등록 폼 진입 |
| `메인 홈으로` | 9장 역할별 메인 (나중에 진행 가능) |

- **가입 완료 ≠ 상세등록 완료**

### 2-6. 만 14세 미만

| 1차 | 내용 |
|-----|------|
| 제외 | 만 14세 미만 **전용 화면·분기** |
| 제외 | 법정대리인 인증 플로우 |
| 운영 | 학생 축 = 학부모 기준 통합 |

## 4. 회원 구조 잠금

### 4-1. 공통 회원 모듈

- 회원가입은 공부방/과외/학부모가 **따로 노는 방식이 아님**.
- **공통 계정** 위에 **역할·프로필**이 붙는 구조.

### 4-2. 다중 역할 · 단일 화면

| 가능 | 불가 |
|------|------|
| `study_room_owner` + `tutor` **동시 보유** | 로그인 후 **역할 2개 화면 동시** |

→ **4장 §6** · 로그인 후 **선택한 역할 1개** 기준 화면만 (9장 연동)

### 4-3. 개념 모델 (1차)

```
공통 회원 (users + user_profiles)
 ├── user_roles (guardian_student | study_room_owner | tutor | admin)
 ├── students (1:N) ── student_subject_targets
 ├── study_rooms (5장)
 └── tutors (4장·DDL TODO)
```

## 6. 공부방 등록 DB 잠금 (개념)

**테이블·컬럼·매핑:** [5장](05-study-room-db.md)

### 6-1. 성격

학부모가 **비교·판단**할 수 있는 **샵 상세 정보** — 1차부터 얕게 두지 않음.

### 6-2. 1차 핵심 축 (잠금)

공부방명 · 운영자명 · 소개문 · 위치 · 교습장 형태(재택/교습소) · 수업 인원 · 주력과목 · 대상 학년/대상군 · 수업 스타일 · 교습비 · 경력 · 모집 정보 · 연락 가능 시간 · 교육청 등록 여부 · 특징 항목 · 사진

### 6-3. 가격

- **숫자 비교형** + **텍스트 설명형** **둘 다** (`price_amount` + `price_description`)

### 6-4. 시설/환경

- **체크형** (`facility_masters` + `study_room_facilities`)
- **자유기술형** (`facility_note`)

### 6-5. 지역 (5장 §7·§9)

- `study_rooms.region_id/complex_id` = **공부방 기본 위치**
- `study_room_regions` = 로그인 후 **지도/핀/메인 노출** 저장 지역 (**최대 3 · 대표 1**)
- 단지 있음 → **단지 우선** · 없으면 **동**(빌라 포함)

### 6-6. 확장 포석

과목/대상 매핑 · 이미지 다건 · 시설 구조화 · 후기 · **광고 슬롯과 정보 분리**
